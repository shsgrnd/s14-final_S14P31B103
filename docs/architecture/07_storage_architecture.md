# 전체 저장소 구조도

## 문서 개요

본 문서는 GitCat MVP에서 사용하는 전체 저장소 구조를 정의한다.

GitCat은 로컬 기반 VS Code Extension으로 동작하며, 서버 없이 **SQLite**, **로컬 파일 시스템**, **VS Code Secrets**를 조합해 데이터를 관리한다.

각 저장소는 데이터 특성에 따라 역할을 분리하며, 메타데이터·관계 데이터는 SQLite에, 실제 스냅샷 파일 본문은 로컬 파일 시스템에, 민감정보는 VS Code Secrets에 저장한다.

---

## 저장소 구성 개요

| 저장소 | 역할 | 저장 대상 |
| --- | --- | --- |
| SQLite | 관계형 메타데이터 저장소 | 세션, 스냅샷 메타데이터, 변경 기록, 원복 이력, 병합 분석, 충돌 후보, AI 추천 이력 |
| 로컬 파일 시스템 | 실제 파일 본문 저장소 | 스냅샷 원본 파일, 병합 초안 파일, 요약 JSON, 임시 작업 파일 |
| VS Code Secrets | 민감정보 저장소 | 사용자 AI API Key |

---

## 1. SQLite 저장소

### 역할

SQLite는 GitCat의 **주 메타데이터 저장소**로 사용한다.

조회, 정렬, 필터링, 관계 추적이 필요한 데이터는 모두 SQLite에서 관리한다.

### 저장 대상

- 세션(Session)
- 스냅샷 메타데이터(Snapshot)
- 변경 기록(ChangeRecord)
- 원복 이력(RestoreHistory)
- 병합 분석(MergeAnalysis)
- 충돌 후보(ConflictCandidate)
- AI 추천 결과(RecommendationHistory)

### 저장 이유

- 세션 ↔ 스냅샷 ↔ 변경기록 ↔ 원복이력 간 관계 조회가 필요함
- 브랜치, reason, 날짜, 체크포인트 여부 기준 필터링이 필요함
- 최근 순, 사용 빈도 순, 원복 횟수 순 등의 정렬/집계 가능성이 높음

### 예시 조회

- 특정 브랜치의 ai_work 스냅샷 목록 조회
- 특정 세션에 연결된 스냅샷과 변경 기록 함께 조회
- 원복이 가장 많이 일어난 스냅샷 top N 조회
- 체크포인트 제외 오래된 스냅샷 자동 정리 대상 조회

---

## 2. 로컬 파일 시스템 저장소

### 역할

로컬 파일 시스템은 GitCat의 **실제 파일 본문 저장소**로 사용한다.

SQLite에는 경로·인덱스·메타데이터만 저장하고, 코드 원문과 같이 용량이 큰 데이터는 파일 시스템에 둔다.

### 저장 대상

- 스냅샷 원본 파일 복사본
- 병합 초안 결과 파일
- 요약 정보 JSON
- 임시 비교/원복용 파일

### 저장 이유

- 실제 코드 파일은 텍스트 크기가 크고 개수가 많을 수 있음
- 파일 단위 복원은 파일 시스템이 더 자연스러움
- SQLite에는 파일 본문 전체보다 경로와 메타데이터만 저장하는 편이 효율적임

### 저장 위치 예시

- `.vscode/gitcat/snapshots/`
- `.vscode/gitcat/merge-sessions/`
- `.vscode/gitcat/temp/`

---

## 3. VS Code Secrets

### 역할

VS Code Secrets는 GitCat의 **민감정보 저장소**다.

### 저장 대상

- 사용자 AI API Key

### 저장 이유

- API 키는 일반 파일이나 SQLite에 평문 저장하지 않기 위함
- VS Code가 제공하는 민감정보 전용 저장소를 활용하기 위함

### 비저장 대상

아래 데이터는 Secrets에 저장하지 않는다.

- 스냅샷 메타데이터
- 세션 정보
- 변경 기록
- 병합 분석 결과
- 추천 이력

---

## 저장 원칙

### 1. 메타데이터와 본문 분리

- 관계형 조회가 필요한 정보 → SQLite
- 실제 파일 원문/복사본 → 파일 시스템

### 2. 민감정보 분리

- API Key → VS Code Secrets
- 일반 메타데이터와 혼합 저장 금지

### 3. 원복 가능성 우선

- 원복에 필요한 실제 원본 파일은 반드시 파일 시스템에 보관
- 해당 파일의 식별자 및 인덱스는 SQLite에서 관리

### 4. 추후 서버 확장 가능성 고려

- SQLite에는 로컬 메타데이터 중심 저장
- 파일 저장 규약은 추후 서버 업로드/동기화 가능하도록 식별자 기반으로 설계

---

## 데이터 저장 흐름

### 1. AI 작업 세션 시작

- SQLite에 Session 생성
- 세션 상태 active 기록

### 2. 변경 파일 추적

- 최초 변경 전 파일 원본을 로컬 파일 시스템에 저장
- SQLite에 관련 스냅샷/세션 메타데이터 기록

### 3. 세션 종료 후 스냅샷 확정

- 파일 시스템에 스냅샷 디렉터리 생성
- SQLite에 Snapshot 메타데이터 저장
- ChangeRecord 생성 및 연결

### 4. 원복 수행

- SQLite에서 대상 스냅샷 및 파일 목록 조회
- 파일 시스템에서 원본 복사본 로드
- 현재 워크스페이스에 복원
- SQLite에 RestoreHistory 저장

### 5. 병합 분석 및 추천

- SQLite에 MergeAnalysis, ConflictCandidate, RecommendationHistory 저장
- 필요 시 파일 시스템에 병합 초안 JSON 저장

---

## 저장소별 책임 요약

| 구분 | 책임 |
| --- | --- |
| SQLite | “무엇이 저장되었는가”와 “서로 어떤 관계인가”를 관리 |
| 파일 시스템 | “실제 어떤 파일 내용을 복원해야 하는가”를 관리 |
| VS Code Secrets | “외부 AI 호출에 필요한 민감 키”를 안전하게 관리 |

---

## 결론

GitCat은 단일 저장소가 아니라, 데이터 성격에 따라 저장소를 분리한다.

SQLite는 관계형 메타데이터를, 로컬 파일 시스템은 실제 파일 본문을, VS Code Secrets는 민감정보를 담당한다.

이 구조를 통해 조회 효율성과 복원 안정성을 동시에 확보한다.
