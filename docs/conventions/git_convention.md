# Git 브랜치 및 커밋 컨벤션 가이드

GitCat 프로젝트의 표준 Git 컨벤션 가이드입니다. 
효율적이고 일관성 있는 협업을 위해 모든 팀원은 아래 규칙을 준수하여 브랜치를 생성하고 커밋을 작성합니다.

## 🌿 브랜치 (Branch) 컨벤션

브랜치 이름은 작업 종류, 간략한 설명, 그리고 지라(Jira) 티켓 번호(코드)를 조합하여 작성합니다.

**형식:** `[작업타입]/[브랜치설명]_[지라코드]`

### 작업 타입 (Type)
- `feat`: 새로운 기능을 추가할 때
- `fix`: 버그를 수정할 때
- `design`: UI/UX 디자인 수정 (CSS 등)
- `refactor`: 코드 리팩토링 (기능 변경 없음)
- `docs`: 문서 작업 (README.md, 기획서 등)
- `test`: 테스트 코드 추가 및 수정
- `chore`: 빌드 업무 수정, 패키지 매니저 설정, CI/CD 등 기타 작업

### 예시
- `feat/login-api_S14P31B103-12` (로그인 API 구현)
- `fix/button-click-error_S14P31B103-15` (버튼 클릭 에러 수정)
- `docs/update-prd_S14P31B103-24` (PRD 리드미 문서 업데이트)

> **주의**: 영문 소문자와 숫자를 사용하며, 띄어쓰기 대신 하이픈(`-`)을 사용합니다. 지라 코드와 브랜치 설명 사이에는 정확히 언더바(`_`)를 사용하여 구분합니다.

## 💬 커밋 (Commit) 메시지 컨벤션

커밋 메시지는 작업 타입과 함께 지라 티켓 번호를 포함하는 것을 권장합니다. 이를 통해 PR 리뷰 시 변경 이력과 지라 요구사항을 쉽게 매핑할 수 있습니다.

**형식:** `[작업타입]: [커밋 요약 및 설명] (#지라코드)`

### 예시
```text
feat: 구글 소셜 로그인 기능 추가 (#S14P31B103-12)

- OAuth 2.0 기반 구글 로그인 연동
- User 엔티티 및 스키마 수정
```

## 🚀 워크플로우 가이드

1. **지라 티켓 생성**: 개발 착수 전, Jira 이슈를 발급받고 작업 코드를 확인합니다.
2. **브랜치 생성**: 최신 `develop` (혹은 `main`) 브랜치에서 위 컨벤션에 맞게 새로운 로컬 브랜치를 생성합니다.
   `git checkout -b feat/task-name_S14P31B103-12`
3. **작업 및 커밋**: 작고 논리적인 단위로 커밋을 진행합니다.
4. **Pull Request 발행**: 
   GitHub에 Push 완료 후 PR을 생성할 때, `.github/pull_request_template.md`가 자동 적용됩니다. 
   템플릿 빈칸을 성실히 채우고 지라 코드 번호를 작성한 뒤 리뷰어를 지정합니다.
5. **리뷰 및 머지**: 최소 1명 이상의 승인(Approve)을 얻은 뒤 타겟 브랜치에 Merge 합니다.
