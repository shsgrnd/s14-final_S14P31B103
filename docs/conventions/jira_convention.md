# Jira 티켓 컨벤션 가이드

GitCat 프로젝트의 표준 Jira 컨벤션 가이드입니다.  
효율적이고 일관성 있는 이슈 관리를 위해 모든 팀원은 아래 규칙을 준수하여 티켓을 생성합니다.

---

## 🗂️ 티켓 구조

Jira 이슈는 **에픽(Epic) → 스토리(Story) → 태스크(Task)** 3단계 계층으로 구성됩니다.

```
Epic
└── Story
    ├── Task
    └── Task
```

---

## 🟣 에픽 (Epic)

에픽은 하나의 큰 기능 또는 도메인 단위를 나타냅니다.  
여러 스토리를 묶는 최상위 단위로, **굵직한 작업 주제**를 기준으로 생성합니다.

**형식:** `[작업 내용]`

### 예시
- `사용자 인증`
- `고양이 프로필 관리`
- `알림 시스템`
- `관리자 대시보드`

> **주의**: 에픽은 범위가 너무 넓거나 좁지 않도록 도메인 또는 기능 모듈 단위로 작성합니다.

---

## 🟡 스토리 (Story)

스토리는 에픽 내에서 사용자 관점의 기능 단위를 나타냅니다.  
하나의 에픽 아래 여러 스토리가 존재할 수 있으며, **스프린트 내 완료 가능한 범위**로 작성합니다.

**형식:** `[작업 내용]`

### 예시
- `구글 소셜 로그인`
- `프로필 이미지 업로드`
- `푸시 알림 수신 설정`
- `회원 목록 조회 및 필터링`

> **주의**: 스토리는 "사용자가 ~할 수 있다"는 관점에서 작성하되, 티켓 제목은 간결하게 기능명으로 표기합니다.

---

## 🔵 태스크 (Task)

태스크는 스토리를 구현하기 위한 실제 작업 단위입니다.  
담당자와 작업 파트(BE/FE/Infra 등)를 명확히 구분하기 위해 **작업파트를 접두어로 표기**합니다.

**형식:** `[작업파트] 작업 내용`

### 작업파트 종류
| 작업파트 | 설명 |
|---|---|
| `BE` | 백엔드 개발 |
| `FE` | 프론트엔드 개발 |
| `Infra` | 인프라 / DevOps |
| `Design` | UI/UX 디자인 |
| `Docs` | 문서 작성 |

### 예시
- `[BE] 구글 OAuth 2.0 로그인 API 구현`
- `[FE] 로그인 페이지 UI 구현`
- `[BE] User 엔티티 및 DB 스키마 설계`
- `[FE] 프로필 이미지 업로드 컴포넌트 개발`
- `[Infra] S3 이미지 업로드 버킷 설정`
- `[Docs] 소셜 로그인 API 명세서 작성`

> **주의**: 작업파트는 대괄호(`[]`)로 감싸고, 이후 작업 내용과 띄어쓰기로 구분합니다. 작업 내용은 명확하고 구체적으로 작성합니다.

---

## 🚀 워크플로우 가이드

1. **에픽 생성**: 스프린트 시작 전, 팀 단위로 에픽을 정의하고 생성합니다.
2. **스토리 작성**: 에픽 하위에 스프린트 내 구현할 기능 단위로 스토리를 생성합니다.
3. **태스크 분배**: 스토리 하위에 파트별 구현 단위로 태스크를 생성하고 담당자를 지정합니다.
4. **개발 착수**: 태스크의 Jira 코드를 확인한 뒤, [Git 브랜치 컨벤션](./git-convention.md)에 따라 브랜치를 생성하고 작업합니다.
5. **상태 관리**: 작업 진행에 따라 태스크 상태를 `할 일 → 진행 중 → 완료`로 업데이트합니다.
