## 📌 기록 모음

### 🗓️ 2026-04-24 GitCat 초기 인프라 설계 방향성 결정
**1. 배경 및 문제 상황**
- MVP 구조 설계 시, 서버를 둘 것인가 아니면 로컬 기반으로 갈 것인가에 대한 논의 진행.

**2. 고려했던 대안들**
- 클라우드 DB & Backend 서버 연동: 확장성은 좋으나 유저의 로컬 소스코드 노출 우려.
- 순수 로컬 기반 (SQLite + File System): 보안 및 반응성 우수, 단 팀 단위 동기화 불가.

**3. 최종 결정 및 논리**
- **결정:** 100% 로컬 기반 Serverless MVP 채택.
- **Why:** GitCat의 핵심 가치는 "AI 작업 단위의 즉각적이고 안전한 원복"과 "보안"임. 코드가 외부에 유출되는 것을 꺼리는 개발자의 니즈를 충족시키기 위해, 속도와 보안 측면에서 로컬 SQLite와 파일 시스템을 직접 제어하는 것이 MVP 목적에 가장 부합한다고 판단함.

**5. 결과 및 배운 점**
- 시스템 아키텍처 문서를 로컬 기반으로 전면 수정함. 추후 서버 확장을 고려하여 메타데이터와 파일 본문 저장소를 분리하는 아키텍처 패턴을 학습함.

### 🗓️ 2026-04-24 로컬 SQLite 데이터베이스 연동 및 DAO 패턴 아키텍처 수립
**1. 배경 및 문제 상황**
- MVP에서 메타데이터 관리를 위해 `packages/storage`에 데이터베이스를 연동하고 스키마를 초기화하는 과정에서 폴더 구조 및 확장성에 대한 고민 발생.

**2. 고려했던 대안들**
- 대안 A: 별도의 `packages/database` 패키지 생성 (의존성 분리는 좋으나, 파일 I/O와 DB가 쪼개져 응집도가 떨어짐)
- 대안 B: `packages/storage/src/sqlite/` 내부에 평탄하게(Flat) 스크립트 작성 (빠르지만 확장이 어려움)
- 대안 C: `client`, `migrations`, `repositories` 도메인 폴더로 분리 (초기 세팅 비용이 들지만 유지보수에 압도적으로 유리)

**3. 최종 결정 및 논리**
- **결정:** 대안 C 채택 및 `better-sqlite3` 연동.
- **Why:** ERD 분석 결과 테이블이 15개에 달할 정도로 관계가 복잡하고, 백엔드 개발자들이 향후 `Session`, `Snapshot` 등 세부 도메인별 쿼리를 작성해야 함. 이를 위해 `repositories` (DAO 패턴) 폴더를 분리하고, `migrations/schema.ts`에서 중앙 집중식으로 DDL을 관리하는 엔터프라이즈급 폴더 아키텍처가 가장 적합하다고 판단함.

**4. 결과 및 배운 점**
- DB 파일의 확장자(`.db` vs `.sqlite`)의 차이가 없음을 검증하고, 프로젝트 명세서에 작성된 `gitcat.db` 컨벤션을 그대로 차용함.
- 초반에 평탄한 구조로 짰다가, 팀이 설계한 도메인 단위 폴더 구조를 재발견하고 즉각적으로 마이그레이션 및 리팩토링하는 경험을 함. 이를 통해 독단적 개발이 아닌, 팀의 아키텍처 의도를 파악하는 것의 중요성을 배움.

### 🗓️ 2026-04-24 AI 파이프라인 연동 및 모노레포 병합 충돌(Conflict) 해결
**1. 배경 및 문제 상황**
- AI 파이프라인, 보안 래퍼(SecretStorage), 익스텐션 진입점(Entrypoint) 등의 인프라 뼈대를 구축한 후 `develop` 브랜치를 병합(Pull)하는 과정에서 충돌 발생.
- `shared-types` 및 `simple-git` 패키지를 사용하는 타 파트의 작업물과 내 작업물(`openai` 등)이 의존성 및 TS 설정에서 부딪힘.

**2. 고려했던 대안들**
- 대안 A: 내 커밋을 되돌리고 남의 코드를 베이스로 다시 작업 (번거롭고 내 작업 유실 가능성 존재)
- 대안 B: 코드 손실 없이 양쪽 의존성을 결합하고, 모노레포 루트 설정을 수정하여 근본적인 TS 빌드 오류 억제

**3. 최종 결정 및 논리**
- **결정:** 대안 B 채택.
- **Why:** 모노레포 구조 특성상 패키지 간 의존성이 빈번하게 얽히므로, `package.json`에서 의존성을 `workspace:*` 패턴으로 병합하고, `tsconfig.base.json`에 `declaration: true`와 `moduleResolution: "node10"` 설정을 강제하여 추후 모든 패키지의 `.d.ts` 누락 문제 및 구버전 경고를 일괄 해결하는 것이 가장 효율적이라 판단함.

**4. 결과 및 배운 점**
- `pnpm install` 후 `pnpm run build` 결과, 4개 패키지 빌드가 단 하나의 타입 에러 없이 성공함. 모노레포 내 `tsconfig.base.json`의 파급력과 pnpm workspace의 패키지 관리 메커니즘을 확실히 이해하게 됨.

### 🗓️ 2026-04-24 `better-sqlite3` 설치 스크립트 승인 경고 확인
**1. 배경 및 문제 상황**
- Windows 환경에서 `pnpm install`을 다시 수행한 뒤 `Ignored build scripts: better-sqlite3@12.9.0` 경고가 출력됨.
- `pnpm run build`, `pnpm run lint`는 정상 통과했지만, SQLite를 실제로 사용하는 런타임 단계에서 문제가 생길 수 있는지 확인이 필요했음.

**2. 고려했던 대안들**
- 대안 A: 경고를 무시하고 진행 (당장 타입 체크와 빌드는 되지만, 실제 DB 연결 시점의 실패 가능성을 뒤로 미룸)
- 대안 B: `pnpm approve-builds`로 `better-sqlite3` 스크립트 승인 여부를 확인하고, 런타임 테스트 전 선제적으로 정리

**3. 최종 결정 및 논리**
- **결정:** 대안 B 기준으로 TODO에 반영하고, 실제 SQLite 런타임 연결 전에 승인 여부를 확인하기로 함.
- **Why:** 이번 경고는 빌드 실패가 아니라 `pnpm v10`의 보안 정책에 따른 스크립트 차단이므로, 정적 검증과 런타임 검증을 구분해서 관리하는 편이 명확함. 특히 `better-sqlite3`는 네이티브 모듈이라 실제 사용 시점에야 문제가 드러날 수 있으므로 인프라 마무리 단계에서 별도 확인이 필요함.

**5. 결과 및 배운 점**
- 현재 기준으로는 TypeScript 빌드와 ESLint는 모두 통과했으며, 남은 리스크는 SQLite 실제 연결 시점의 런타임 검증으로 좁혀짐.
- `pnpm v10`에서는 네이티브 모듈 설치가 예전처럼 자동으로 되지 않을 수 있으므로, 설치 성공과 실행 가능 여부를 분리해서 확인해야 한다는 점을 정리함.
